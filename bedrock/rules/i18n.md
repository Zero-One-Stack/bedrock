# Rule: Internationalization (i18n)

> **Non-negotiable.** No hardcoded user-facing strings — ever (this is a hard ban in `CLAUDE.md`).
> All copy goes through **react-i18next**; layout works in any locale and in RTL. This rule is the
> depth behind the ban; `typescript-and-quality.md` just points here.

## The baseline

- **Every user-facing string** comes from a translation key via `useTranslation` — or, for a pure
  presentational atom, is passed in as a prop (the consumer translates). Never inline literal copy.
- Keys live in **namespaced** locale files; keep `en` (or the default) and every other locale **in
  sync** — a missing key is a bug, not a silent fallback to the key string.

## Key naming & structure

- **Namespace by FSD layer + slice** so the i18n tree mirrors the codebase. Format:
  `<layer>.<slice>.<key>`. Shared keys (button labels, errors that don't belong to a slice)
  live under `shared.common.*`.
  - `shared.common.{save, cancel, retry}` — kit-wide reusable strings.
  - `entities.employee.{statusActive, statusInactive}` — entity-domain strings.
  - `features.fileGrievance.{submit, fileNew, success, failure}` — action strings.
  - `widgets.grievanceDashboard.{title, emptyState}` — widget composition strings.
  - `pages.activeGrievances.{title, metaDescription}` — route screen strings.
  - Load only the namespaces a route needs; the slice owns its namespace file
    (`locales/<locale>/features/file-grievance.json`).
- Keys are **semantic, not the English text**: `customer.profile.saveButton`, not
  `"Save profile"`. Renaming copy then doesn't churn keys.
- Group by component/intent; keep hierarchy shallow (2–3 levels).

## Interpolation, plurals, formatting

- **Interpolate**, don't concatenate: `t('greeting', { name })` → `"Hello {{name}}"`. Never build
  a sentence by joining translated fragments — word order differs across languages.
- **Pluralize with i18next plural keys** (`_one`/`_other`, ICU where configured) — never an
  `if (n === 1)` in the component.
- **Format numbers/dates/currency** with `Intl` (via i18next formatters), locale-aware — never
  hand-format. Ties to any Zod/date handling in `services-and-data.md`.

## Layout, RTL & a11y

- Layout must survive **text expansion** (German/Finnish run long) — no fixed-width labels that
  truncate; let content flow (`responsive-design.md`).
- **RTL:** use CSS **logical properties** (`margin-inline`, `padding-inline`, `inset-inline`) and
  `dir`-aware tokens — not `left`/`right`. Set `dir` from the locale. (The kit's CSS-logical
  styling already leans this way — keep it.)
- `lang`/`dir` set on `<html>` per the active locale; screen readers and hyphenation depend on it.

## Next.js App Router

- Use the project's chosen i18n routing (sub-path `/[lng]/…` or domain), configured once; record
  it in `project-specifics.md`. Load namespaces server-side where possible; hydrate the client.
- Don't ship every locale's full bundle to every user — split by locale/namespace.

## Testing

- Render through `test-utils` (it wraps `I18nextProvider`) so `t()` resolves in tests
  (`testing.md`). Assert on **rendered output**, not raw keys.
- It's reasonable to test that a key renders *something* (no raw-key leakage) and that plural/
  interpolated forms pick the right branch.

## Hard rules

- ❌ Any hardcoded user-facing string (the hard ban) — including in errors, `aria-label`s,
  placeholders, toasts, and E2E-visible text.
- ❌ String-concatenating translated fragments; manual pluralization; hand-formatted dates/numbers.
- ❌ `left`/`right` physical CSS for anything that must mirror in RTL — use logical properties.
- ❌ Keys named after English copy; out-of-sync locale files.
- ✅ Semantic namespaced keys; interpolation + i18next plurals + `Intl` formatting; logical CSS;
  `lang`/`dir` per locale.
